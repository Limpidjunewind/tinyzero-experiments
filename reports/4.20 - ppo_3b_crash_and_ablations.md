# PPO 3B 训练崩溃：现象、消融、指向根因

> 更新日期：2026-04-20
>
> 本报告合并了原 `4.18 entropy_coeff_ablation_3B` 与 `4.20 entropy_collapse_analysis`。

---

## 故事开头

前一份 batch size 消融实验显示：把 batch 从 64 提高到 128 能稳定触发 Aha Moment，test score 从 ~0.44 提升到 ~0.57。顺着这个方向，我们把 batch 继续提高到 256，希望学习信号更干净、Aha Moment 更早出现。

结果跑 bs=256 系列时碰到了意外：训练中途模型会突然开始输出 `<think>!!!!!!!!!!</think>`，reward 归零。同一超参，不同 run 崩溃步数差几百步，也有 run 跑到 step 400+ 不崩。

本报告记录三件事：**崩溃的观测**、**NaN 传播的数学机制**、**四组超参消融的结果**。第五节把这些事实汇总，得出一个指向"训推不一致"的假设。对这个假设的直接验证由 [4.24 train_infer_mismatch.md](./4.24%20-%20train_infer_mismatch.md) 继续。

## 二、崩溃的数值机制

崩溃的直接原因是 NaN 污染权重。按照 PPO 的 loss 计算，NaN 传播链是：

```
某个 token 的 log_prob = NaN
 → ratio = exp(new_log_prob − old_log_prob) = NaN
 → loss 的 mean 里混入 NaN → loss = NaN
 → 梯度全部 = NaN
 → 更新后权重全部 = NaN
 → 模型输出退化
 → 下一步梯度继续 = NaN
```

loss 的 `mean` 把一个 NaN 扩散到所有参数；权重变成 NaN 之后梯度也是 NaN，自我恢复的路径被堵死。

社区已报告的同形态现象：

| Issue | 现象 |
|-------|------|
| [verl #721](https://github.com/volcengine/verl/issues/721) | `old_log_prob` NaN → kl_loss NaN → 输出 `!!!` |
| [verl #628](https://github.com/volcengine/verl/issues/628) | 训练中突然只生成 `!!!!!!!` |
| [verl #747](https://github.com/volcengine/verl/issues/747) | NaN → reward 归零 → 重复 `!` |
| [TinyZero #63](https://github.com/Jiayi-Pan/TinyZero/issues/63) | Qwen2.5-3B，8×A800，输出全 `!` |

## 三、NaN 从哪来：两条数学上可能的路径

PPO ratio：

```
ratio = exp(new_log_prob − old_log_prob)
```

verl 的 hybrid engine 下：
- `old_log_prob` 由 **vLLM** 生成阶段产生（PagedAttention，bf16 原生）
- `new_log_prob` 由 **FSDP** 训练阶段重算（Flash Attention 2，fp32 master + bf16 autocast）

两条路径在同样的权重和 token 上可能算出不同的 log_prob（社区记录：[verl #2088](https://github.com/volcengine/verl/issues/2088)、[#2996](https://github.com/volcengine/verl/issues/2996)、[#2305](https://github.com/volcengine/verl/issues/2305)）。

在这个设定下，NaN 有两条数学可能：

### Path A：两边同时下溢

```
old_log_prob = −inf, new_log_prob = −inf
→ exp(−inf − (−inf)) = exp(NaN) = NaN
```

触发条件：某个 token 的概率极低，两个引擎都下溢为 `-inf`。

### Path B：训推不一致，只有一边下溢

```
old_log_prob = −inf（vLLM 下溢）
new_log_prob = −87.9（FSDP 未下溢）
→ exp(−87.9 − (−inf)) = exp(inf) = inf → 后续产生 NaN
```

触发条件：两个引擎对同一个 token 给出的 log_prob 差异足够大，使得其中一个下溢、另一个未下溢。

### 崩溃前后的 wandb 数据

bs256-mini128 首次崩溃（Job 589226）崩溃前后：

| step | entropy_mb0 | entropy_mb1 | kl_mb0 | kl_mb1 | grad_norm | score |
|------|-------------|-------------|--------|--------|-----------|-------|
| 88 | 0.2203 | 0.2315 | 0 | 0.0004 | 0.1005 | 0.4695 |
| 89 | 0.2086 | 0.2143 | 0 | 0.0002 | 0.1032 | 0.4105 |
| **90** | 0.2181 | **NaN** | 0 | **NaN** | **NaN** | 0.4262 |
| 91+ | NaN | NaN | NaN | NaN | NaN | 0.0000 |

- NaN 首次出现在 step 90 的 **mb1**，mb0 仍正常。
- `kl_mb0` 全程为 0。机制上：mb0 在第一次参数更新之前计算，old_log_prob 和 new_log_prob 走的是同一条 FSDP forward，所以 KL 精确为 0。
- 崩溃前一步 score 0.4695，无下降预兆。

## 四、消融实验：四组对照

### 4.1 实验设计

思路是用两个常用的稳定性超参（entropy bonus、KL 约束）来尝试抑制崩溃。

**四组的算法变量**

| 组 | 实验名 | `entropy_coeff` | `kl_coef` |
|---|---|---|---|
| **Baseline** | `ppo-3b-bs256-mini128-monitor`（重跑）| 0.001 | 0.001 |
| **Ent-01** | `ppo-3b-entropy001` | 0.01 | 0.001 |
| **Ent-05** | `ppo-3b-entropy005` | 0.05 | 0.001 |
| **KL-05** | `ppo-3b-kl005` | 0.001 | 0.05 |

**四组的硬件/并行差异**

| 参数 | Baseline | Ent-01 / Ent-05 / KL-05 |
|---|---|---|
| `tensor_model_parallel_size` | 2 | 1 |
| `gpu_memory_utilization` | 0.4 | 0.25 |
| `ref.param_offload` | false | true |

三组硬件差异只影响显存分配和切分，不改变梯度计算逻辑。

**四组共享的超参**

| 类别 | 设置 |
|---|---|
| 模型 | Qwen/Qwen2.5-3B（base + critic 同权重） |
| 数据 | countdown，`train_batch_size=256`，`max_prompt_length=256`，`max_response_length=512`，`val_batch_size=1312` |
| PPO / Actor | `ppo_mini_batch_size=128`，`clip_ratio=0.2`，actor lr=1e-6，`ppo_epochs=1`，`grad_clip=1` |
| Critic | `ppo_mini_batch_size=128`，`cliprange_value=0.5`，lr=1e-5 |
| 算法 | GAE（`gamma=1`, `lam=1`），`kl_penalty=kl`（fixed），`use_kl_loss=false` |
| Rollout | vLLM，bf16，`temperature=1`，`top_p=1`，`top_k=-1`，`n=1`，`enforce_eager=true` |
| 其他 | `enable_gradient_checkpointing=true`，`use_dynamic_bsz=true`，`total_epochs=15` |
| 硬件 | 2×H100-96 |

Baseline 与第一节首次崩溃的 bs256-mini128 run 超参相同，属于重跑。

### 4.3 生成行为采样

#### 训练初期（step 1，四组均为未经 RL 的 base model 行为）

**Baseline（step 1）**：

```
<think> I need to make 91 * 97 and then subtract 63 or 75 from it.
I can divide 91 by 63 to get the closest, but we can't use 63 yet. ... </think>
<answer> (91/97) * 75 + (91//97)*75 = 23 </answer>   → Invalid
```

**Ent-01（step 1，来自老 4.18 报告）**：

```
<think> Considering the numbers [37, 6, 27, 3], we have different
multiplication and addition possibilities... (6 * 11) + 6 = 67. </think>
<answer> (6 * 11) + 6 </answer>   → Invalid（使用了不存在的数字 11）
```

**Ent-05（step 1，来自老 4.18 报告）**：

```
<think> Given the numbers [38, 55, 26, 52], we need to create an equation
that equals 61... </think>
<answer> (55 - 38)  + 26 </answer>   → Invalid
```

**KL-05（step 1）**：

```
<think>First, I'll find the sum of the numbers: 9 + 37 + 51 + 77 = 174</think>
Now, I need to divide this sum by 20 and obtain a whole number...
[长段自问自答式重复尝试]
```

四组初期行为一致：`<think>` 段有自然语言推理尝试，会列多种运算但逻辑混乱、计算错误频繁，输出格式不稳定（经常出现 `Invalid equation` 或 `No equation found`）。

#### 训练晚期

**Baseline（step 200）**：

```
<think> We need to use the numbers 87, 44, 49, 17 exactly once to get 58.
Let's try different combinations: (87-49)+44-17=65, too high.
(87-44)+49-17=75, too high. ... (44*17)-(87-49)=710, too high. </think>
<answer>(87 - 44) - (49 - 17)</answer>   → Wrong: 11

<think> We need to use 38, 68, 38 to get 67. (68-38)+38=68, too high.
(68-38)-38=-8, too low. ... </think>
<answer>(68 - 38) + 38</answer>   → Wrong: 68

<think> (88-39)+9=49+9=58... This works! </think>   → Correct ✓
<think> (2*71)-69=142-69=73, that's it! </think>   → Correct ✓
```

格式稳定，`<think>` 内容是系统地枚举加减乘除组合，简单题能答对（~0.62 test score）。

**Ent-01（step ~400，来自老 4.18 报告）**：

```
<think>42 + 23 =65   - 14 =51 </think>
<answer>(42+23-14)</answer>   → Correct: 51

<think>85-60=25, Then 25 adder pronto fatten stuff again unrest...</think>
<answer>(85 - 60 +42+ 6)</answer>   → Wrong
```

`<think>` 内容为加减法计算混合乱码文本，未出现乘除。

**Ent-05（step 76，崩溃态）**：

```
<think><|endoftext|>
```

直接输出结束符。

**KL-05（step 200）**：

```
<think> 83 is the largest number, so we can subtract it from another number
to reduce it. Subtract 69 from 83 to get 14. Now, we have three numbers left:
35, 14, and 2. Adding 14 to 2 gives us 16. To get 45, we need to multiply 16
by 3, so we can use the remaining 35 for that. </think>
<answer> (83 - 69) + (14 + 2) * 3 = 45 </answer>   → Invalid

<think>First, I need to use 98... subtract it from 98 gives 41. Now we need
to make 41 using 6 and 47. As 47 divided by 6 is approximately 7.83... </think>
<answer> (47 - (98 / 6))</answer>   → Wrong: 30.67

<think> We can start by trying to come up with a way to apply the operations... </think>
Now, let's add 50 - 34 to 16 to reach the final answer of 50. ...
<answer> 78 - 44 + 16 </answer>   → Correct ✓
```

`<think>` 有完整的自然语言推理，包含加减乘除尝试，但格式常不遵守（`<answer>` 标签后追加散文），且经常引入原题未给的数字导致 Invalid。

## 五、把观察串起来

基于前四节的数据，下面是本报告推出的结论。

**1. 崩溃是概率性事件，不是超参错误。**

第一节的三次崩溃和第四节重跑的 Baseline 超参相同，结果一个群体崩、一个没崩。说明崩溃的触发条件不在超参配置本身，而在训练过程中的某种偶发数值事件。

**2. Ent-01 和 KL-05 两组"没崩"是通过限制策略更新实现的。**

从曲线走势直接可读：
- Ent-01 的 entropy 被抬在 ~0.5-1.5，response_length 稳定在 ~50-60（低于 Baseline 的 260，也低于其他组），test score 0.45 停在平台。
- KL-05 的 entropy 从头到尾几乎不动（~1.0），response_length 也几乎不动（~200-220），test score 只到 0.22。

两组都没崩，但也都付出了明显的学习能力代价。

**3. Ent-05 表明"加大 entropy 系数"不是越大越安全。**

`entropy_coeff` 从 Ent-01 的 0.01 升到 Ent-05 的 0.05 后，entropy 飙至 ~10，step 75 崩溃，test score 不动。调整方向相同、数值更大，并未单调提升稳定性。

**4. 三组崩溃时的 entropy 水平与两条路径的匹配度。**

Path A 要求某个 token 的概率极低，两边都下溢——通常对应 entropy 已经很低的阶段。第一节数据：灰色组崩溃时 entropy ~0.4-0.5，比粉 ~0.3、绿 ~0.2 都高。这一点与 Path A 的触发条件不完全吻合；Path B 不要求 entropy 到特定水平，只要两个引擎 log_prob 分歧足够大即可。本报告数据更倾向 Path B，但未直接测量两边 log_prob，不构成证明。

**综合这四点**，本报告的推论是：崩溃的根因不在 `entropy_coeff` 或 `kl_coef` 的数值，而在 `ratio` 计算所依赖的 `old_log_prob` 和 `new_log_prob` 不来自同一条前向路径。验证这个推论需要直接测量两个引擎 log_prob 的差异，这部分见 [4.24 train_infer_mismatch.md](./4.24%20-%20train_infer_mismatch.md)。

## 六、本报告没有回答的问题

1. NaN 具体来自 Path A 还是 Path B 未直接验证。本报告只有"三组崩溃时 entropy 差异大"这一间接线索，未在 ratio 计算前分别观测两边 log_prob。
2. 崩溃是概率性的，但触发频率未量化。目前只观察到"同一超参有 run 崩、有 run 不崩"，未系统统计崩溃率随 step 的分布。
3. 三组没崩的消融都伴随学习能力下降。本报告数据无法分离"防崩"和"压学习"这两件事是否本质相关。
