# Entropy Coefficient 消融实验：Qwen2.5-3B PPO

> **实验背景**：在 batch_size 消融实验中确认 batch=256 会导致 entropy collapse（模型输出 `!!!`，reward hacking）。本实验测试 entropy bonus（`entropy_coeff`）是否能通过鼓励输出多样性来阻止 collapse。

---

## 一、实验设计

### 对比组

| | 实验名 | entropy_coeff | 其余配置 |
|--|--------|--------------|---------|
| **A（基线）** | `ppo-3b-batchtest` | 0.001 | 同 |
| **B** | `ppo-3b-entropy001` | 0.01 | 同 |
| **C** | `ppo-3b-entropy005` | 0.05 | 同 |

三组除 `entropy_coeff` 外，所有训练超参完全一致。

> **注**：组 A（ppo-3b-batchtest）的硬件配置与 B/C 略有差异（`tensor_model_parallel_size=2`，`gpu_memory_utilization=0.4`，`ref.param_offload=False`），B/C 均为 `tensor_parallel=1`，`gpu_util=0.25`，`ref.param_offload=True`。这些参数只影响显存分配，不影响梯度计算逻辑。

### 验证问题

提高 `entropy_coeff` 能否阻止 batch=256 下的 entropy collapse？如果能，模型的学习行为（reward、response length、推理质量）会有什么变化？

---

## 二、实验配置

三组共同超参：

| 参数 | 值 |
|------|----|
| 基座模型 | Qwen/Qwen2.5-3B |
| `train_batch_size` | 256 |
| `ppo_mini_batch_size`（actor / critic） | 128 |
| updates per rollout | 2（= 256 / 128） |
| `kl_coef` | 0.001 |
| `clip_ratio` | 0.2 |
| actor `lr` | 1e-6 |
| critic `lr` | 1e-5 |
| `max_response_length` | 512 |
| `total_epochs` | 15 |
| GPU | 2 × H100-96 |

各组差异：

| | A | B | C |
|--|---|---|---|
| `entropy_coeff` | 0.001 | 0.01 | 0.05 |
| 实际训练步数 | ~57 | ~400 | ~77 |

## 四、生成行为对比

#### 训练初期（step 1 附近）

三组初期表现相似，均为未经训练的 base model 行为：`<think>` 中有长段自然语言推理尝试，会尝试多种运算但逻辑混乱、计算错误频繁，输出格式不稳定。

**A 组（step 1）**：
```
<think> We have three numbers: 22, 16, and 41. We can try combining them
using basic arithmetic operations to get a result of 79... It seems like
there is no combination... </think>
<answer> The numbers [22, 16, 41] cannot be combined... </answer>
```

**B 组（step 1）**：
```
<think> Considering the numbers [37, 6, 27, 3], we have different
multiplication and addition possibilities... (6 * 11) + 6 = 67.</think>
<answer> (6 * 11) + 6 </answer>   → Invalid（使用了不存在的数字 11）
```

**C 组（step 1）**：
```
<think> Given the numbers [38, 55, 26, 52], we need to create an equation
that equals 61... </think>
<answer> (55 - 38)  + 26 </answer>  → Invalid
```

#### 训练晚期

**A 组（step 54-57，崩溃态）**：
```
<think>!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!</think>
```
模型输出被 `!` 填满，无任何算术内容。所有题目 `No equation found`。

**B 组（step ~400，训练末段）**：
```
<think>42 + 23 =65   - 14 =51 </think>
<answer>(42+23-14)</answer>   → Correct: 51

<think>85-60=25, Then 25 adder pronto fatten stuff again unrest...</think>
<answer>(85 - 60 +42+ 6)</answer>   → Wrong: 73≠61
```
`<think>` 中有真实加减法计算，但混杂乱码文本。只使用加减法，未出现乘除。简单题能答对，难题答错。

**C 组（step 76-77，崩溃态）**：
```
<think><|endoftext|>
```
模型直接输出结束符，完全停止生成。所有题目 `No equation found`。

## 六、结论

**1. entropy_coeff=0.01 能阻止 entropy collapse，entropy_coeff=0.05 不能。**

- A 组（0.001）：entropy 降至 ~0（step 50），test score 归零，输出 `!!!` — collapse。
- B 组（0.01）：entropy 维持 ~0.5-1.5（step 75-330），test score 从 ~0.15 升至 ~0.45 — 存活。
- C 组（0.05）：entropy 飙至 ~10（step 30），grad norm 飙至 ~2.4，test score 归零 — 反向崩溃。

entropy_coeff 过低不防 collapse，过高导致梯度被 entropy 项主导（entropy 项贡献 ~0.5 vs pg_loss ~0.01-0.1）。0.01 在两者之间，训练存活。

**2. entropy_coeff=0.01 阻止了 collapse，但模型只学到加减法，无 Aha Moment。**

- test score 稳定在 ~0.45，但 response length 固定在 ~50-60，无先降后升的变化。
- 日志中 `<think>` 全部是线性加减（如 `42+23=65, 65-14=51`），未出现乘除或回溯。
- 简单题（加减直接凑出）答对，难题（需要乘除组合）答错。

**3. B 组训练末段（step 330+）出现不稳定信号。**

- entropy_loss 从 ~0.5-1.5 上升至 ~3-4。
- 日志中 `<think>` 开始混杂乱码文本（如 `25 adder pronto fatten stuff again unrest...`）。
- test score 在此阶段仍维持 ~0.45，尚未下降，但 entropy 上升趋势已经出现。
