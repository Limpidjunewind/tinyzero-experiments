# PPO 训练突然崩溃：从正常到全输出 `!!!`

> 更新日期：2026-04-20

---

## 1. 现象

训练过程中所有指标正常，某一个 step 突然模型开始输出 `<think>!!!!!!!!!!!!!!!!</think>`，reward 归零，无法恢复。崩溃是概率性的：同样的超参，有时 step 50 崩，有时 step 400+ 都不崩。崩溃前没有预警信号。

以下三组实验均观察到崩溃：

| 颜色 | 实验 | 崩溃 step |
|------|------|----------|
| 🩷 粉 | ppo-3b-bs128-mini16 | ~75 |
| 🟢 绿 | ppo-3b-bs256-mini128-monitor | ~90 |
| 🩶 灰 | ppo-3b-bs128-mini128 | ~130 |

#### Response Length

![response_length](../assets/ppo%20-%20entropy-collapse-analysis/reponse-length-mean.png)

崩溃时 response length 跳至 512（max_response_length 上限）。崩溃前三组均在 100-300 范围波动。

#### Critic Score

![critic_score](../assets/ppo%20-%20entropy-collapse-analysis/critic-score-mean.png)

三组崩溃前 critic score 均在上升（粉色峰值 ~0.5，绿色 ~0.45，灰色 ~0.35），崩溃后立即归零。

#### Actor Entropy Loss

![entropy](../assets/ppo%20-%20entropy-collapse-analysis/actor-entropy-loss.png)

三组 entropy 均持续下降，但下降速度和崩溃时的 entropy 水平差异大：粉色崩溃时 ~0.3，绿色 ~0.2，灰色 ~0.4-0.5。崩溃后 entropy 均跌至底部。

---

## 2. 崩溃机制

**因果链**：log_prob 出现 NaN → ratio = NaN → loss = NaN → 梯度全 NaN → 权重全 NaN → 模型输出退化 → 无法恢复。

一个 token 的 NaN 足以通过 loss 均值传播到所有参数。权重变 NaN 后，每一步的梯度也是 NaN，所以永远无法恢复。

**社区证据**：[verl #721](https://github.com/volcengine/verl/issues/721) 记录了完整的 old_log_prob → NaN → kl_loss → NaN → 输出 `!!!` 链条。

---

## 3. NaN 的来源：两条路径

PPO 的 ratio 计算 `ratio = exp(new_log_prob - old_log_prob)` 中，`old_log_prob` 由 vLLM 生成，`new_log_prob` 由 FSDP 计算。

**路径 A：两边都下溢**

```
old_log_prob = -inf, log_prob = -inf
→ exp(-inf - (-inf)) = exp(NaN) = NaN
```

**路径 B：训推不一致，只有一边下溢**

vLLM 和 FSDP 用不同的 kernel 和浮点运算顺序，对同一 token 算出不同结果：

```
old_log_prob = -inf, log_prob = -87.9
→ exp(-87.9 - (-inf)) = exp(inf) = inf → 后续产生 NaN
```

社区已报告 vLLM/FSDP 之间的 log_prob 不一致（[verl #2088](https://github.com/volcengine/verl/issues/2088)、[verl #2996](https://github.com/volcengine/verl/issues/2996)、[verl #2305](https://github.com/volcengine/verl/issues/2305)）。

**从现有数据无法区分是哪条路径。** 要区分需要在 ratio 计算前分别检查两个 log_prob 的值。

另外，也存在 kernel 层面的数值错误：[verl #2656](https://github.com/volcengine/verl/issues/2656)（kernel fusion 后 logprobs 出错）、[verl #2899](https://github.com/volcengine/verl/issues/2899)（Triton kernel vocab size 整除问题）。

## 4. 本地实验数据

### 4.1 bs256-mini128 monitor 实验（Job 589226）

配置：bs=256, mini=128, entropy_coeff=0.001, 2×H100。step 91 开始全输出 `!!!`。

崩溃前后关键指标（wandb）：

| step | entropy_mb0 | entropy_mb1 | kl_mb0 | kl_mb1 | grad_norm | score |
|------|-------------|-------------|--------|--------|-----------|-------|
| 88 | 0.2203 | 0.2315 | 0 | 0.0004 | 0.1005 | 0.4695 |
| 89 | 0.2086 | 0.2143 | 0 | 0.0002 | 0.1032 | 0.4105 |
| **90** | **0.2181** | **NaN** | **0** | **NaN** | **NaN** | **0.4262** |
| 91+ | NaN | NaN | NaN | NaN | NaN | 0.0000 |

观察：
- NaN 首次出现在 step 90 的 mb1（第二个 mini-batch），mb0 正常
- kl_mb0 在整个训练中始终为 0（mb0 是第一个 mini-batch，模型尚未被更新，old_log_prob 和 new_log_prob 应一致）
- score 在崩溃前仍在上升（step 88 达到 0.4695），无性能下降预兆

### 4.2 entropy_coeff 消融

| entropy_coeff | 结果 |
|--------------|------|
| 0.001（默认） | ~step 50 崩 |
| 0.01 | 存活至 step 400+ |
| 0.05 | ~step 30 崩，entropy 反向飙升 |

每组只跑了 1 次，无法排除运气。

### 4.3 三组崩溃时的 entropy 差异

三组崩溃时的 entropy 跨度为 0.2-0.5（粉 ~0.3，绿 ~0.2，灰 ~0.4-0.5）。如果崩溃由 token 概率极端低引起（路径 A），应在 entropy 更低时才发生。灰色在 entropy 仍较高时就崩溃了，与路径 B（训推不一致）更一致，但不构成证明。

## 6. 已知缓解方法

| 方法 | 状态 |
|------|------|
| 提高 `entropy_coeff`（0.001 → 0.01） | 本地单次实验有效，未多 seed 验证 |
| FP16 代替 BF16 | 论文推荐，未本地验证 |
| Rollout Correction（IS/RS） | verl 已实现，未本地验证 |
| `log_prob` clipping | 社区报告有效，未本地验证 |
