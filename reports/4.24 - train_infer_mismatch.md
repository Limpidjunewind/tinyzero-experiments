# TinyZero 训练中的「!!!」崩溃：训推不一致问题

## 一句话结论

**症状**：3B Qwen 在 countdown 上做 PPO，训练一段时间后 response 退化为一长串 `!!!`。

**定位**：排除 reward hacking、entropy collapse、数值溢出之后，看到 verl 在 hybrid engine 下做了一个选择——vLLM 采样之后用 FSDP 重新 forward 算 `old_log_probs`。这让 PPO 第一个 micro-batch 的 `ratio ≡ 1`（维持 PPO 推导里的起点不变量），代价是 π_old ≠ 实际采样分布，梯度估计带一个 deterministic 的 bias。

**验证**：把 `old_log_probs` 换成 vLLM 原生的之后，指纹指标 `kl_mb0` 从恒等于 0 变成 ~10⁻³，和实测的 FSDP-vs-vLLM log_prob 差异一致；观测窗口（step 0-280）内 `!!!` 未再出现，test score 收敛到 ~0.60，与 baseline 同一水平。

---

## 一、怎么发现的

最初的假设一条条被排除：

1. **Reward hacking？** 不是。`!!!` 并不能拿到正奖励，reward 曲线也没出现「作弊」特征。
2. **Entropy collapse？** 看起来像，但 entropy 下降是*结果*不是*原因*——要解释为什么会 collapse 到 `!!!` 这个特定的无意义 token。
3. **数值溢出/NaN？** wandb 上 grad_norm、pg_loss、ppo_kl 所有曲线都是**平滑**的，没有任何突变点。如果是溢出应该是一个尖峰然后崩，但曲线是「可以预测的」缓慢漂移。

关键观察 → 曲线平滑但结果错误 = **系统性 bias**，不是随机故障。系统性 bias 最可能来自算法公式的某个假设被悄悄破坏。PPO 里最容易被破坏的假设就是 **π_old = 实际采样分布**。

于是怀疑：**训推不一致**。

## 二、如何验证

### 2.1 查代码

在 `verl/workers/rollout/vllm_rollout/vllm_rollout.py:216`，找到决定性的一行：

```python
# 'old_log_probs': log_probs,   # we will recompute old log prob with actor
```

vLLM 在生成 token 的时候其实已经顺手把每个 token 的 log_prob 算出来了。但作者**注释掉了**，不用它。

然后在 `verl/workers/fsdp_workers.py:425-435`，rollout 结束后，FSDP 会**重新跑一遍 forward**，用自己算的 log_prob 当作 `old_log_probs` 塞回 batch。

- vLLM 的 forward：PagedAttention kernel，bf16 原生
- FSDP 的 forward：Flash Attention 2 kernel，fp32 master weights + bf16 autocast

**同样的权重、同样的 token 序列，走两条不同的 kernel/精度路径 → log_prob 必然有差异。**

### 2.2 Smoke 测试

为了不浪费集群时间，先写一个最小验证脚本 `smoke_test_logprob_mismatch.py`：

**设计**：
1. 加载 Qwen2.5-3B base
2. 用 vLLM 生成若干 prompt 的续写，同时记录 vLLM 自己的 logprobs
3. 把同样的 (prompt + response) 喂给 HuggingFace + Flash Attention 2，重新算一遍 log_prob
4. 对比两组 log_prob

**结果**：
- 单 token log_prob 最大绝对差：**0.11 ~ 0.16**
- 转成 PPO ratio（`exp(diff)`）：最大 **1.08x ~ 1.17x**

### 2.3 smoke 能证明什么，不能证明什么

smoke 有两个局限：

1. **用的是 HF，不是 FSDP**。真实训练里重算 log_prob 的是 FSDP（fp32 master + bf16 autocast + Flash Attention 2）。HF 直接 bf16 加载 + Flash Attention 2 只是 FSDP 的代理：kernel 相同，但 FSDP 多一层 all-gather 和 mixed precision wrapper。要完全对齐需要直接在 FSDP worker 里测（2.5）。
2. **只证明了「两个引擎给出不同 log_prob」，没证明「这个差异导致了 `!!!`」**。两个独立命题：
   - 命题 1：vLLM 和 FSDP 算出的 log_prob 不一样 ← smoke 直接证实
   - 命题 2：这个差异就是 `!!!` 崩溃的根因 ← smoke 没证

命题 1 从「猜」变成「事实」。命题 2 需要控制实验：改掉 `vllm_rollout.py:216` 和 `fsdp_workers.py:425`，用 vLLM 原生 log_prob，同超参跑一次，看 `!!!` 是否消失。这部分在第四节。

### 2.4 粗略估计 1.17x 的影响

PPO 的 clip 阈值是 ±0.2（ratio ∈ [0.8, 1.2]），1.17x 初始 ratio 已经吃掉大半 clip budget，但仍在 clip 范围内。单次梯度估计的 bias 不大，能否积累出 `!!!` 崩溃这一步仍属推断。真正要靠 2.5 的实测 + 第四节的控制实验来判断。

### 2.5 在真实 FSDP 上测量：Method A

Smoke 测试只是 HF 代理。为消除这个不确定性，直接在 verl 里加 instrumentation：

- `vllm_rollout.py`：把 vLLM 算出的 log_probs 额外存到 batch（key：`vllm_log_probs`）
- `fsdp_workers.py` 的 `compute_log_prob`：recompute 完 FSDP log_probs 之后，计算 `diff = fsdp_lp - vllm_lp`，逐 step 写到 `mismatch_live.jsonl`

跑了 50 步 PPO（3B Qwen on countdown），结果：

| 指标 | 典型值 | 峰值 |
|---|---|---|
| `lp_abs_mean` | ~0.018 | — |
| `lp_abs_max` | ~0.37 | 0.69 |
| `ratio_max` | ~1.4 | **1.99** |
| `ratio_gt_1p2` | ~0.1% | — |

比 HF smoke 的 1.17x 更大——真实 FSDP 路径上 ratio 峰值接近 2x。可能的原因：FSDP 多了 fp32 master + bf16 autocast + all-gather，数值差比 HF 直接 bf16 更大。

至此「两个引擎 log_prob 系统性不同」从 smoke 推断变成直接观测。

## 三、作者的设计与它的代价

作者注释掉了 vLLM 的 log_probs，改用 FSDP 重算。这是有意的选择，不是 bug。下面分两步看清这个选择本身，以及它在 hybrid engine 下的后果。

### 3.1 PPO 的一个不变量

标准 PPO 的推导里：
- 收集一批样本，记下采样时的策略，叫 π_old
- 做 K 轮梯度更新，每轮计算 ratio = π_new / π_old
- 第一个 micro-batch，actor 还没更新 → π_new 和 π_old 是同一套参数 → ratio ≡ 1，KL ≡ 0

`ratio_mb0 = 1` 不是凑出来的，是公式本身的恒等式。这个不变量成立的前提是：**算 π_old 的那次 forward，和实际采样时用的 forward，是同一条路径**。

经典单引擎 PPO（只有 FSDP，不用 vLLM）里，这个前提天然成立。

### 3.2 Hybrid engine 打破了这个前提

verl 把生成和训练拆到两个引擎：
- 采样用 vLLM（PagedAttention，bf16）
- 训练用 FSDP（Flash Attention 2，fp32 master + bf16 autocast）

所以「实际产生这些 token 的分布」是 vLLM 的分布，和 FSDP forward 算出的分布不是同一个东西（Method A 实测 ratio 最大 1.99x）。

这时候 `old_log_probs` 只能二选一：

| 选项 | old_log_probs 来源 | 维持的性质 | 失去的性质 |
|---|---|---|---|
| A（作者）| FSDP recompute | `ratio_mb0 ≡ 1`，PPO 公式的起点不变量 | π_old ≠ 实际采样分布 |
| B（fix） | vLLM 原生 | π_old = 实际采样分布 | `ratio_mb0 ≠ 1`，第一步就是 off-policy |

两边只能取一个。作者取 A。

### 3.3 取 A 的代价

PPO 目标函数：

```
L = E_{x ~ π_old} [ min( r(x) · A,  clip(r(x), 1-ε, 1+ε) · A ) ]
```

期望下标 `x ~ π_old` 要求：x 是从 π_old 采样的。选项 A 下：
- x 实际来自 vLLM 分布
- π_old 被定义成 FSDP 重算的分布
- 两者不是同一个分布

所以梯度估计有一个 deterministic 的 bias（vLLM 和 FSDP kernel 差异固定），每步同向累积。这个 bias 是否足以让模型最终退化到 `!!!`，从公式本身推不出来，要靠第四节的控制实验判断。

### 3.4 取 B 的代价

选项 B 保住了 π_old = 实际采样分布，但 `ratio_mb0` 不再是 1。PPO 从第一个 micro-batch 起就是真实 off-policy：
- clip 从第一步起就可能触发
- pg_loss 方差比 A 大（ratio × advantage 的乘积方差更大）
- 学习速度可能更慢（见 response_length 图，aha 出现得更晚）

B 的代价换来的是 π_old 和采样分布一致，importance sampling 的前提成立。

## 四、控制实验：修复后看到了什么（Method C）

2.5 证明了 mismatch 真实存在。还差一步：**去掉 recompute、直接用 vLLM log_probs，`!!!` 会消失吗？**

### 4.1 控制变量

两个 run 对照：

| | Baseline | Fix |
|---|---|---|
| `experiment_name` | `ppo-3b-bs256-mini128-monitor` | `ppo-3b-fix-vllm-lp` |
| `old_log_probs` 来源 | FSDP recompute | vLLM 原生 |
| `vllm_rollout.py:216` | 注释掉 vLLM log_probs | 启用 vLLM log_probs |
| `fsdp_workers.py` recompute 块 | 运行 | gate 跳过 |

除上述代码 patch，其余超参完全一致：

| 类别 | 设置 |
|---|---|
| 模型 | Qwen/Qwen2.5-3B（base + critic 同权重） |
| 数据 | countdown，`train_batch_size=256`，`max_prompt_length=256`，`max_response_length=512`，`val_batch_size=1312` |
| PPO / Actor | `ppo_mini_batch_size=128`，`clip_ratio=0.2`，actor lr=1e-6，`entropy_coeff=0.001`，`ppo_epochs=1`，`grad_clip=1` |
| Critic | `ppo_mini_batch_size=128`，`cliprange_value=0.5`，lr=1e-5 |
| 算法 | GAE（`gamma=1`, `lam=1`），`kl_coef=0.001`（fixed），`kl_penalty=kl`，`use_kl_loss=false` |
| Rollout | vLLM，bf16，`temperature=1`，`top_p=1`，`top_k=-1`，`n=1`，`tensor_model_parallel_size=2`，`gpu_memory_utilization=0.4`，`enforce_eager=true` |
| 其他 | `enable_gradient_checkpointing=true`，`use_dynamic_bsz=true`，`total_epochs=15` |
| 硬件 | 2×H100-96 |

两条曲线的系统性差异只能归因于 code patch。

### 4.2 `kl_mb0`：直接对应 3.2 的二选一

verl 把每次 PPO update 的第一个 micro-batch 单独打指标（`mb0`）。

![kl_mb0](../assets/ppo%20-%20%20train_infer_mismatch/kl-mb0.png)

| | Baseline（选项 A）| Fix（选项 B）|
|---|---|---|
| `kl_mb0` | 恒等于 0（平直线） | 0-0.0015 波动，多数 ~0.0003-0.0008 |

基线 `kl_mb0 ≡ 0` 直接对应 3.1 里的不变量：π_old 和 π_new 走同一条 FSDP forward，KL 精确为 0。

Fix 的 `kl_mb0 > 0`，量级和 Method A 里测到的 FSDP-vs-vLLM log_prob diff 一致——这就是把 `old_log_probs` 换成 vLLM 后 PPO 公式里第一眼能看到的变化。

### 4.4 证据分层

| 命题 | Smoke（HF） | Method A（真实测量） | Method C（fix run） |
|---|---|---|---|
| 两个引擎 log_prob 不同 | 推断 ratio_max 1.17x | 实测 ratio_max 1.99x | — |
| 差异在 PPO 公式里可观测 | — | — | `kl_mb0` 从 ≡0 变成 ~10⁻³ |
| 修复后 `!!!` 不再出现 | — | — | step 0-280 grep 0 次 |
| 修复不牺牲学习能力 | — | — | test score 收敛 ~0.60，与 baseline 同水平 |

Baseline 本身是第一节里会概率性崩溃的 bs256-mini128 配置，只是这一次 run 没崩（[4.20 报告](./4.20%20-%20ppo_3b_crash_and_ablations.md)）。单次 fix run 与单次成功的 baseline 在 test score 上收敛到同一水平，未观察到 fix 降低学习能力。但"fix 能阻止概率性崩溃"这一命题需要多 seed 重复才能确证。

## 五、修复方向

最小修改：用 vLLM 原生 log_probs。

```python
# vllm_rollout.py:216
'old_log_probs': log_probs,   # 直接用 vLLM 的
```

```python
# fsdp_workers.py:425  跳过重算
if ... and False:   # 禁用 recompute
    ...
```

注意 `meta_info['temperature']` 等赋值若原本在 recompute 块里，要移出来——`update_actor` 下游依赖。

社区参考：[sail-sg/Precision-RL-verl](https://github.com/sail-sg/Precision-RL-verl) 针对同一问题做了 fp16 路径的修复。

## 六、本报告没有回答的问题

1. **Fix 能否阻止概率性崩溃未确证**。Baseline 本身就可能不崩（见 4.20 §1），单次 fix run 跑通不能证明 fix 改变了崩溃率。需要多 seed 重复统计。
2. **观测窗口到 step 280**。已知的三个崩溃样本分布在 step 75-130，fix run 已越过该窗口；但更长训练下是否出现新的失效模式未测。
