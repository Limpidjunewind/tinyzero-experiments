# TinyZero Reward 设计对比实验报告

## 1. 项目概述

### 1.1 TinyZero 项目简介

TinyZero 是 [DeepSeek R1 Zero](https://github.com/deepseek-ai/DeepSeek-R1) 的复现项目，基于 [veRL](https://github.com/volcengine/verl) 框架构建，使用 PPO 强化学习在 Countdown 推理任务上训练小规模语言模型（Qwen2.5-3B），使其自主发展出推理和自我验证能力。

### 1.2 Countdown 任务

Countdown 任务要求模型使用给定的一组数字（3~4 个）和四则运算（+, -, *, /），组合出等于目标值的等式。每个数字只能使用一次。

**示例：**

> **输入：** 使用 [4, 8, 2, 39] 组合出 29
>
> **期望输出：** `<think>39 - 8 - 2 = 29, 没有用到 4... 试试 (39 - 8) - 4 + 2 = 29</think><answer>(39 - 8) - 4 + 2</answer>`

### 1.3 训练框架

训练使用 veRL 框架的 PPO 实现，核心流程为：

```
Prompt → Actor 生成 response → Ref Policy 计算 KL → Critic 估计 V(s)
    → Reward 打分 → GAE 优势估计 → Critic 更新 → Actor PPO 更新
```

所有组件通过 Ray 进行分布式调度，采用 FSDP 进行数据并行。

### 1.4 Reward 设计的关键性

在 PPO 训练中，reward function 是唯一的外部监督信号。它的设计直接决定了：
- 模型能学到什么能力（正确率、推理方式、运算复杂度）
- 模型如何学（收敛速度、探索效率）
- 训练本身的效率（response 长度影响计算开销）

本实验围绕 **reward function 的设计** 展开，在相同训练配置下对比三种不同的 reward 方案，考察其对模型推理能力的影响。

## 2. 三种 Reward 方案设计

### 2.1 v1：离散 Outcome Reward（Baseline）

v1 是 [TinyZero 原始项目](https://github.com/Jiayi-Pan/TinyZero)自带的 reward 实现，采用三档离散打分，奖励仅作用在序列最后一个 token 上（Sparse ORM）：

| 情况 | 得分 |
|------|------|
| 无法提取 `<answer>` 标签 | 0 |
| 格式正确但答案错误 | 0.1 |
| 完全正确 | 1.0 |

**设计特点：** 简单直接，但奖励粒度粗——"差 1" 和 "差 100" 的错误答案得到相同的 0.1 分，模型无法获得"越接近越好"的梯度信号。

**v1 训练日志暴露的问题：**

训练 15 epochs 后，对 v1 的输出采样统计（1/64 采样，共 8296 条）：

| 类别 | 占比 | 问题 |
|------|------|------|
| 答对 | 34.2% | — |
| 数字对但算错 | 58.5% | 这些全部得 0.1 分，但偏差从 1 到 100+ 不等 |
| 数字用错 | 5.3% | 同上 |
| 格式不对 | 2.2% | 得 0 分 |

其中，58.5% 的"算错"样本的偏差分布为：≤5 占 16.4%，6~10 占 12.3%，11~20 占 19.3%，>20 占 52.0%。差 1 的答案和差 100 的答案得到完全相同的分数。

此外，正确答案中仅 **3.3%** 使用了乘除法；错误答案的 `<think>` 平均 1086 字符，远长于正确答案的 520 字符，说明模型在错误方向进行了大量无效探索。

### 2.2 v2：多维连续 Outcome Reward

在 v1 基础上引入 6 个评分维度，将离散奖励改为连续信号（Continuous ORM）：

| 维度 | 规则 | 分值 |
|------|------|------|
| 格式分 | 同时有 `<think>` 和 `<answer>` | 0.05 |
| 数字使用分 | 给定数字恰好使用一次 | 0.05 |
| **接近度分** | `0.4 × max(0, 1 - |result-target|/|target|)` | **0 ~ 0.4** |
| 完全正确奖励 | 结果正确且数字全部正确 | +0.5 |
| 运算复杂度加分 | 使用了 * 或 /（仅答对时） | +0.1 |
| 自我验证加分 | `<think>` 中包含验证行为（仅答对时） | +0.05 |

**总分范围：** 0.0 ~ 1.2

**核心改进——接近度分：** 提供连续梯度信号。例如 target=100 时，result=99 得 0.396 分，result=50 得 0.2 分，result=200 得 0 分。模型可以从"接近正确"的答案中获得正向反馈，而非只在完全正确时才有奖励。

**设计动机：** 基于 v1 训练日志分析发现：58.5% 的错误答案（wrong result）全部获得相同的 0.1 分，但偏差范围从 1 到 100+ 不等；正确答案中仅 3.3% 使用乘除法；错误答案的 think 平均 1086 字符（远长于正确答案的 520 字符），说明模型在错误方向大量无效探索。

### 2.3 v3：Outcome Reward + Process Reward Model (PRM)

在 v2 的 outcome reward 基础上，额外引入一个 **Process Reward Model (PRM)**，对 `<think>` 内的每个中间算术步骤单独打分（Dense Process Reward）：

```
总 reward = v2 outcome reward（最后 token） + PRM 过程奖励（每个中间步骤 token）
```

**PRM 模型：** 手搓的 Encoder-Only Transformer，所有组件（Multi-Head Attention、FFN、LayerNorm）从头实现，不使用 PyTorch 封装层。

| 超参数 | 值 |
|--------|-----|
| d_model | 256 |
| n_heads | 8 |
| n_layers | 6 |
| d_ff | 1024 |
| vocab | 字符级（~22 个字符：数字、运算符、括号等） |
| 总参数量 | **~4.8M** |

**任务定义：** 输入一个算术声明（如 `80 - 35 = 45`），输出该声明是否正确的概率（二分类）。

**PRM 训练数据：** 使用程序自动生成 100,000 条训练 / 10,000 条测试数据。正样本为正确的算术等式，负样本为对正确结果施加随机扰动后的错误等式。数据覆盖四则运算，复杂度从 2 个数到 5 个数嵌套括号不等。

**PRM 自身准确率：** 在测试集上达到 **66.2%**（二分类，随机基线 50%）。准确率不算高，但足以为 PPO 提供有方向性的过程奖励信号。

**集成方式：**
1. 从 response 的 `<think>` 中用正则提取所有形如 `expr = number` 的中间步骤
2. PRM 预测每步正确概率 p
3. 在对应 token 位置注入 `prm_weight × (2p - 1)` 作为过程奖励
   - p > 0.5 → 正向奖励（鼓励正确步骤）
   - p < 0.5 → 负向惩罚（惩罚错误步骤）
4. PRM 运行在 CPU 上（模型较小），不占用 GPU 显存

**prm_weight = 0.1**（相对 outcome reward 的 ~1.0 较小），避免过程奖励压过最终答案的评判。

**设计动机：** v2 虽然改善了 outcome reward 的粒度，但奖励仍然只在序列末尾。`<think>` 中的中间推理步骤无论对错都不影响 reward。根据 RL 理论，dense reward 比 sparse reward 能更高效地指导策略学习（Ng et al., 1999），OpenAI "Let's Verify Step by Step"（2023）也证明 PRM 在数学推理中优于 ORM。

### 2.4 三方案对比总结

| | v1 | v2 | v3 |
|---|---|---|---|
| **Reward 类型** | Sparse / 离散 | Sparse / 连续 | Dense / 连续 |
| **奖励位置** | 仅最后 token | 仅最后 token | 最后 token + 中间步骤 |
| **分值范围** | {0, 0.1, 1.0} | [0, 1.2] | [0, 1.2] + 过程奖励 |
| **额外模型** | 无 | 无 | PRM (~4.8M 参数, 66.2% acc) |
| **错误答案梯度** | 无（统一 0.1） | 有（接近度连续） | 有 + 步骤级反馈 |

## 3. 实验设置

三组实验使用完全相同的训练配置，**唯一变量是 reward function**：

| 配置项 | 值 |
|--------|-----|
| 基础模型 | Qwen2.5-3B |
| 硬件 | 4 × NVIDIA L40S (46GB) |
| 训练算法 | PPO (veRL 框架) |
| train_batch_size | 256 |
| total_epochs | 15 (~1920 steps) |
| actor learning rate | 1e-6 (constant) |
| critic learning rate | 1e-5 (constant) |
| KL 系数 | 0.001 (fixed) |
| max_response_length | 1024 tokens |
| 数据集 | Countdown-Tasks-3to4（32768 训练 / 1024 测试） |

## 4. 实验结果与分析

### 4.1 核心结果

| 指标 | v1 | v2 | v3 |
|------|-----|-----|-----|
| **最终验证正确率** | 34.3% | **60.4%** | **62.3%** |
| Wrong result 平均偏差† | 34.7 | 3.4 | **1.1** |
| Wrong result 中偏差=0 占比† | - | 75.5% | **88.8%** |
| 乘除法使用率（正确答案） | 3.3% | 0% | 0% |
| 自我验证出现率 | 极低 | ~0% | ~0% |
| 平均 response 长度（tokens）** | - | **284** | **158** |
| response max（中位数）** | - | **1024**（打满上限） | **220** |
| 训练总时长 | ~27h | ~35.3h | **~26.5h** |
| 每步平均耗时 | - | ~65s | **~46s** |

> † **Wrong result 偏差**：指模型输出的等式计算结果与目标值之间的绝对差。**偏差=0** 表示模型算出了正确的目标数值，但使用了给定集合之外的数字（如题目给 [4, 8, 2, 39]，模型却使用了 3 和 10），因此仍被判为错误。
>
> ** **Response 长度**数据来自 wandb 记录的全训练过程指标（response_length/mean, response_length/max），覆盖所有训练 step 的完整 batch。v1 未记录该指标。

### 4.2 分析一：Outcome Reward 改进是主要贡献

```
v1 (34.3%)  ──[+26.1pp]──>  v2 (60.4%)  ──[+1.9pp]──>  v3 (62.3%)
              连续 ORM 改进                PRM 过程奖励
              (贡献 93%)                   (贡献 7%)
```

从离散到连续的 outcome reward 改进（v1 → v2）贡献了 **93% 的正确率提升**，而 PRM 过程奖励（v2 → v3）仅带来边际提升（+1.9pp）。这表明在当前任务上，**outcome reward 的质量比 reward 的密度更重要**。

接近度分是 v2 最关键的设计——它将"完全正确才有奖励"变为"越接近越好"，为模型提供了连续的优化方向。

### 4.3 分析二：模型算术能力已很强，瓶颈在约束遵循

v3 中 88.8% 的"错误"答案实际上**算出了正确的目标数值**，只是使用了给定集合之外的数字。v2 中这一比例也达到了 75.5%。

这意味着经过 v2/v3 训练后，模型的**算术计算能力已经非常强**，核心瓶颈转移到了"严格使用给定数字"这一约束上。模型倾向于"凑数"——引入不存在的数字来补齐等式，而非纯粹使用给定的数字。

**启示：** 后续改进应聚焦于"数字使用正确性"维度的奖励权重，而非继续提升算术准确性。

### 4.4 分析三：乘除法的"加减法捷径"

三组实验中乘除法使用率均极低（v1: 3.3%, v2/v3: 0%）。模型发现纯加减法就能达到 60%+ 的正确率，没有动力冒险使用乘除法。

v2 虽然设计了运算复杂度加分（+0.1），但该加分仅在完全正确时才生效，且相对接近度分（0.4）和正确奖励（0.5）占比太小，不足以激励模型探索乘除法。

**启示：** 要提升乘除法使用率，可能需要更强的激励（如将复杂运算加分提升至 0.3+），或在训练数据中增加必须使用乘除法才能解答的题目。

### 4.5 分析四：Reward 设计对训练效率的隐性影响

v2 与 v3 使用完全相同的 PPO 超参数并行训练，但 v2 每步耗时 ~65s，v3 仅 ~46s（**v2 慢 41%**）。这一差异是反直觉的：v3 额外运行了 PRM 推理，理应更慢。

**从 wandb 提取的各阶段耗时分解揭示了真正原因：**

| 训练阶段 | v2 (秒) | v3 (秒) | 差异 | 占总差异% |
|---------|---------|---------|------|----------|
| **Rollout 生成** | **30.4** | **13.5** | **+16.9** | **89%** |
| Actor 更新 | 10.1 | 8.3 | +1.8 | 9% |
| Critic 更新 | 17.8 | 17.9 | -0.1 | 0% |
| Ref Policy | 2.0 | 1.5 | +0.5 | 2% |
| Value 估计 | 4.5 | 4.6 | -0.1 | 0% |
| **总计** | **64.8** | **45.8** | **+19.0** | — |

速度差异 **89% 来自 Rollout 生成阶段**（30.4s vs 13.5s，v2 慢 2.25x）。Actor/Critic 更新阶段的耗时几乎相同。

**根因是 response 长度的尾部分布差异：**

| Response 长度 | v2 | v3 |
|--------------|----|----|
| 均值 (tokens) | 284 | 158 |
| 最大值的中位数 | **1024**（打满上限） | **220** |

v2 的平均 response 虽然只是 v3 的 1.8 倍，但 v2 每个 batch 中几乎总有序列打满 1024 token 上限（max 的中位数 = 1024）。vLLM 批量自回归生成时，**整个 batch 必须等最长序列完成**，因此 v2 的极端长尾直接拖慢了整个生成阶段。v3 的 max 中位数仅 220 tokens，batch 完成得更快。

v2 response 更长的原因是其 reward 设计鼓励详细推理（格式分、验证加分等），模型倾向于生成冗长的 `<think>` 内容。v3 的 PRM 通过对中间步骤直接打分，引导模型走向精炼推理，response 更短且更紧凑。

**发现：** Reward 设计不仅影响模型的最终能力，还通过改变输出长度分布间接影响训练效率。这一效应主要作用于 Rollout 生成阶段，且是由 **长尾分布**（而非均值）驱动的。

### 4.6 分析五：PRM 的价值体现在哪里

虽然 PRM 对正确率的边际提升有限（+1.9pp），但它在其他维度展现了价值：

| 维度 | PRM 的贡献 |
|------|-----------|
| 错误答案质量 | 偏差从 3.4 降至 1.1，偏差=0 比例从 75.5% 升至 88.8% |
| 训练效率 | 引导精炼推理，每步快 40%，总时长少 9 小时 |
| 训练稳定性 | 密集奖励信号更稳定，方差更小 |

PRM 的主要作用不是大幅提升正确率，而是**提升错误答案的质量和训练效率**。

## 5. 结论

### 5.1 主要结论

1. **连续 Outcome Reward 是最关键的改进**：从 v1 的离散打分改为 v2 的连续接近度评分，正确率从 34.3% 跃升至 60.4%（+26.1pp），贡献了 93% 的提升。这验证了 reward 信号质量对 RL 训练的决定性作用。

2. **PRM 过程奖励带来边际提升但改善训练效率**：v3 正确率 62.3%，仅比 v2 高 1.9pp，但训练效率提升 40%，且错误答案质量更优。PRM 的价值更多体现在训练过程而非最终指标。

3. **模型瓶颈已从"算术能力"转移到"约束遵循"**：88.8% 的错误答案算出了正确数值但使用了非法数字，说明后续改进应聚焦数字使用约束。

4. **Reward 设计的隐性效率影响不可忽视**：v2 因鼓励详细推理导致 response 变长，训练慢 40%。Reward 设计需同时考虑"学到什么"和"学的效率"。

### 5.2 后续方向

| 方向 | 具体思路 |
|------|---------|
| 强化数字使用约束 | 提高"数字使用分"的权重（如从 0.05 提升至 0.2+） |
| 激励乘除法 | 设计必须使用乘除的训练题目，或大幅提升运算复杂度加分 |
| PRM 消融实验 | 调整 prm_weight（当前 0.1），探索最优过程奖励强度 |
| GRPO 对比 | 尝试 Group Relative Policy Optimization，去掉 Critic 模型 |
